# ChronoTrace 系统逻辑边界说明

> 日期：2026-06
> 范围：重新梳理 ChronoTrace 的系统定位、核心模型、实体身份体系、导入与变更链路，以及哪些能力属于系统核心、哪些能力应保持在边界之外。
> 状态：边界说明草案，用于后续需求取舍、bug 修复排序和 P3.2 方向重审。
> 重要说明：本文件不是实现计划，不要求立即开发新功能；它优先用于防止系统继续向错误方向扩张。

## 1. 一句话定位

ChronoTrace 是一个面向现实业务实体的时态业务台账系统。

它要解决的核心问题不是“怎么做一张可编辑表格”，而是：

```text
现实世界里某个业务实体
    ↓
在系统中有一个稳定锚点
    ↓
它在不同时间点有不同状态
    ↓
每一次状态变化都可审批、可追溯、可回滚、可审计
```

更完整地说：

```text
ChronoTrace = 动态业务表 + 稳定实体 + 时态快照 + 变更批次 + 权限审计
```

它不应该在当前阶段扩张成：

- 万能低代码数据库
- ERP
- CRM
- HR 主数据系统
- 交易流水系统
- 复杂工作流系统
- BI 报表平台
- 多租户 SaaS 基座

ChronoTrace 可以承载这些系统中的一部分“台账型数据”，但不应该替代这些系统本身。

## 2. 核心问题对比

| 问题                  | 容易走偏的理解      | 正确边界                              | 影响                      |
| ------------------- | ------------ | --------------------------------- | ----------------------- |
| 系统到底管什么             | 管所有表格、所有业务流程 | 管“长期存在、可唯一识别、会随时间变化”的业务实体台账       | 决定功能是否纳入核心              |
| 一行数据是什么             | 普通数据库行       | 某个业务实体在某个时间点的状态快照                 | 决定是否需要 `TemporalRecord` |
| identity 是什么        | 用户看着顺眼的名称    | 外部数据对齐同一个 Entity 的稳定匹配码           | 决定导入是否可靠                |
| `business_code` 是什么 | 展示名或标题       | `schema + business_code` 下的实体匹配锚点 | 决定历史是否连续                |
| `display_code` 是什么  | 新的实体 ID      | 用户界面展示码，不参与唯一匹配                   | 防止展示体验污染匹配逻辑            |
| 人员表怎么处理             | 平台内置人员主数据    | 人员只是现实实体的一类，由具体 schema 承载         | 防止 ChronoTrace 变成 HR 系统 |
| 无稳定身份怎么导入           | 系统自动猜测合并     | 阻断、诊断、要求用户提供稳定标识或显式确认             | 防止静默错配                  |
| Excel 导入是什么         | 直接覆盖当前表      | 生成一批 ChangeSet 草稿，经过提交/审批后写入时态记录  | 保留追溯链路                  |

## 3. 核心领域模型

ChronoTrace 当前核心模型可以稳定表达为：

```text
DataSchema
    定义一类业务表，例如资产台账、合同台账、人员授权台账。

Entity
    表示这张表里一个稳定业务对象，例如某台设备、某份合同、某个账号授权。

TemporalRecord
    表示某个 Entity 在一段有效时间内的状态快照。

ChangeSet
    表示一次变更批次，例如一次 Excel 导入、一次批量调整、一次手工修改。

ChangeEntry
    表示 ChangeSet 中针对某个 Entity 的新增、修改或终止。

AuditLog
    记录谁在什么时候做了什么，尤其是敏感操作和关键变更。
```

对应当前实现：

| 模型               | 当前实现位置                              | 职责                                                              |
| ---------------- | ----------------------------------- | --------------------------------------------------------------- |
| `DataSchema`     | `backend/apps/schemas/models.py`    | 动态业务表定义，包含 `fields_config`、`identity_field_key`、时态模式、审批开关和权限可见性 |
| `SchemaVersion`  | `backend/apps/schemas/models.py`    | 记录字段结构版本，使结构演化可追踪                                               |
| `Entity`         | `backend/apps/temporal/models.py`   | 稳定实体锚点，按 `schema + business_code` 唯一                            |
| `TemporalRecord` | `backend/apps/temporal/models.py`   | 实体状态快照，使用 `valid_from / valid_to` 表达有效期                         |
| `ChangeSet`      | `backend/apps/changesets/models.py` | 变更批次，支持 draft、submitted、approved、applied、rejected、reverted      |
| `ChangeEntry`    | `backend/apps/changesets/models.py` | 单条新增、修改、终止操作                                                    |
| `AuditLog`       | `backend/apps/audit/models.py`      | 操作审计，记录 actor、action、target、detail、敏感标记                         |

这组模型的核心价值是：

```text
不是只保存“现在是什么”，而是保存“从什么时候开始是什么、谁改的、为什么改、能不能回看和撤销”。
```

## 4. 实体身份体系

### 4.1 `Entity.id`

`Entity.id` 是系统内部数据库主键。

它适合：

- 作为外键关联 `TemporalRecord`
- 作为 `ChangeEntry.entity_id`
- 作为内部锁定、查询、权限和生命周期操作的稳定引用

它不适合：

- 出现在外部 Excel 里作为业务匹配依据
- 让用户长期记忆
- 作为跨系统业务编号

### 4.2 `business_code`

`business_code` 是现实业务数据进入系统后，用来找到同一个 `Entity` 的稳定匹配码。

当前系统的唯一性约束是：

```text
schema + business_code 唯一确定一个 Entity
```

这意味着：

```text
资产台账 + A-001 -> 一个资产 Entity
合同台账 + A-001 -> 另一份合同 Entity
```

同一个 `business_code` 可以出现在不同 schema 中，但不能在同一个 schema 内代表两个不同实体。

从当前实现看，`business_code` 不应被当作普通可变字段。`ChangeSet` 更新链路会阻止实体 identity 被修改；如果业务上某个编号会变化，应把它作为普通展示字段或状态字段，而不是 identity。

### 4.3 `identity_field_key`

`identity_field_key` 是 schema 配置项，用来说明哪一个字段生成 `business_code`。

单字段示例：

```text
identity_field_key = asset_no
asset_no = A-001
business_code = A-001
```

复合标识兼容方案中，系统使用隐藏字段：

```text
identity_field_key = __identity_code
identity_field_keys = [company_code, employee_no]
business_code = C01|E001
```

这个设计的目的不是让用户直接编辑 `__identity_code`，而是在不破坏 `Entity`、`TemporalRecord`、`ChangeEntry` 现有外键结构的前提下支持复合身份。

### 4.4 `display_code`

`display_code` 是用户界面展示码。

它适合：

- 当前视图第一列
- 生命周期标题
- 变更摘要
- 审批详情
- 导入预览
- 导出展示列
- 搜索结果标题

它不适合：

- 参与 `schema + business_code` 唯一约束
- 作为导入匹配依据
- 作为外部系统长期对齐键

`display_code` 当前通过 resolver 运行时生成，而不是落库字段。这个策略保持了核心模型稳定，也避免展示模板变化立即引发数据库迁移。

### 4.5 身份体系结论

| 概念                   | 解决的问题             | 是否用户可见    | 是否参与唯一匹配 |
| -------------------- | ----------------- | ---------:| --------:|
| `Entity.id`          | 系统内部引用            | 一般不直接展示   | 否        |
| `business_code`      | 外部数据如何找到同一个实体     | 可作为兼容字段展示 | 是        |
| `identity_field_key` | schema 中哪个字段生成匹配码 | 配置层可见     | 间接参与     |
| `__identity_code`    | 复合标识兼容落点          | 默认隐藏      | 是        |
| `display_code`       | 用户如何看懂这条记录        | 是         | 否        |

## 5. 标准数据流

### 5.1 新表 Excel 接入

新表接入的标准链路是：

```text
上传 Excel
    ↓
扫描 sheet 和表头
    ↓
推断 schema draft
    ↓
用户确认字段、identity、时态模式、审批设置
    ↓
解析每一行 data_after
    ↓
resolver 生成 business_code 和 display_code
    ↓
identity 诊断：空值、重复、组合字段错误、系统行号风险
    ↓
通过后创建 DataSchema 和 SchemaVersion
    ↓
生成 Excel 来源的 ChangeSet 草稿
    ↓
用户提交 / 审批 / 应用
    ↓
写入 Entity 和 TemporalRecord
```

关键边界：

- 新表接入不是直接写当前数据。
- 它先生成 schema，再生成 ChangeSet 草稿。
- 只有 ChangeSet 应用后，时态记录才真正生效。

### 5.2 已有表 Excel 导入

已有表导入的标准链路是：

```text
上传 Excel
    ↓
字段映射
    ↓
按 schema resolver 解析 business_code
    ↓
与指定 at 日期的当前有效快照对比
    ↓
生成 create / update / unchanged / invalid
    ↓
根据 missing_policy 生成 keep / terminate
    ↓
生成 ChangeSet 草稿
    ↓
提交 / 审批 / 应用
    ↓
写入新的 TemporalRecord 或关闭旧记录
```

这里最关键的是：

```text
导入不是覆盖表，而是生成一批待确认的时态变更。
```

### 5.3 手工编辑

手工编辑的标准链路是：

```text
用户在当前视图编辑
    ↓
系统创建或复用当前用户的 draft ChangeSet
    ↓
每个新增、修改、终止写成 ChangeEntry
    ↓
提交草稿
    ↓
如果 schema 开启审批，进入 submitted
    ↓
审批通过后 applied
    ↓
如果不需要审批，提交后直接 applied
    ↓
写入 TemporalRecord
```

因此，手工编辑也不应绕过 ChangeSet。

### 5.4 ChangeSet 应用

应用 ChangeSet 的语义是：

| Action      | 对时态记录的影响                           |
| ----------- | ---------------------------------- |
| `create`    | 为 Entity 创建一个新的 `TemporalRecord`   |
| `update`    | 关闭或替代当前有效记录，再创建新的 `TemporalRecord` |
| `terminate` | 关闭当前有效记录，不创建新的状态快照                 |
| `revert`    | 生成反向 ChangeSet，标记原记录被替代或恢复旧状态      |

这里区分两个时间：

| 时间           | 含义                   |
| ------------ | -------------------- |
| `valid_from` | 业务时间，从哪一天开始有效        |
| `applied_at` | 事务时间，系统什么时候真正应用了这批变更 |

ChronoTrace 的价值之一就在于这两个时间可以分离：允许补录过去，也允许提前登记未来变更。

## 6. 系统适用边界

### 6.1 高适配场景

ChronoTrace 适合这些业务：

| 场景              | 为什么适合                     |
| --------------- | ------------------------- |
| 固定资产、设备、工器具台账   | 有稳定编号，状态、位置、责任人会变化，生命周期重要 |
| 软件许可证、账号权限台账    | 授权、责任人、状态和有效期变化需要审计       |
| 合同、证照、资质台账      | 有编号、状态、到期日、附件和审批痕迹        |
| 供应商、客户、合作方主数据   | 准入状态、评级、负责人、风险等级会变化       |
| 项目、任务、里程碑状态台账   | 阶段、负责人、风险、预算状态需要回看        |
| 风险、问题、整改、审计发现台账 | 状态流转、责任归属和证据附件重要          |
| 周期性 Excel 清单    | 需要比较本期和上期差异，并生成可审计变更      |

共同特征：

```text
一行代表一个长期存在的业务实体。
实体有稳定身份。
字段会随时间变化。
历史状态有业务价值。
变更需要审批、回滚或审计。
```

### 6.2 中适配但要控制边界的场景

| 场景      | 可以做什么                 | 不应该做什么              |
| ------- | --------------------- | ------------------- |
| 内部低代码表单 | 扁平字段、标准校验、简单公式        | 复杂动态表单、子表嵌套、跨字段复杂规则 |
| 轻量审批    | 一层提交、审批、驳回            | 多级审批、会签、加签、条件路由、SLA |
| 轻量附件台账  | 合同扫描件、证照图片、整改证据       | 文档管理系统、全文索引、在线批注    |
| 简单引用    | 引用另一个 schema 的 Entity | 主子表、多对多、复杂级联、关系图谱   |
| 简单统计    | 趋势、数量、枚举分布            | BI、透视表、复杂指标、经营驾驶舱   |

### 6.3 低适配或不建议承载的场景

ChronoTrace 当前不适合直接承载：

- 订单明细
- 支付流水
- 库存出入库流水
- 设备遥测
- 接口日志
- 行为埋点
- 财务凭证
- 工资计算
- 考勤绩效
- 复杂采购、销售、生产、BOM
- 多级审批公文流
- 大型 BI 报表
- 强移动端离线采集
- 对外多租户 SaaS 强隔离业务

原因不是这些场景“不重要”，而是它们的核心模型不是 ChronoTrace 当前这套：

```text
schema + entity + temporal snapshot + changeset
```

它们通常需要：

- append-only event stream
- 主从单据
- 交易一致性
- 复杂状态机
- 多角色流程节点
- 数据库侧大规模查询和聚合
- 专门的权限、租户和隔离模型

这些不应该在当前阶段被塞进 ChronoTrace 核心。

## 7. 人员相关能力的边界重审

### 7.1 关键纠偏

人员是现实实体的一种，但 ChronoTrace 不应该内置成人员主数据系统。

也就是说：

```text
可以有人员台账 schema。
可以有账号权限台账 schema。
可以有人员授权、岗位任命、外包人员清单。

但不应该在平台核心中直接引入 PersonMaster、apps.people、person_code 专属映射体系。
```

原因：

1. ChronoTrace 的核心对象是通用 `Entity`，不是 `Person`。
2. 现实实体不仅包括人员，还包括资产、合同、证照、供应商、账号、项目、风险事项等。
3. 如果先内置人员匹配，很容易把通用身份治理写死成 HR 领域规则。
4. 人员匹配规则高度业务化，错配成本高，不适合作为平台默认自动逻辑。

### 7.2 `person_code` 的正确理解

`person_code` 可以作为某个 schema 里的普通业务字段，也可以作为某张人员表的 `identity_field_key`。

但它不应该被理解为平台全局内置身份。

正确理解：

```text
在“人员台账”这个 schema 中：
person_code 可以是一个稳定业务编号。

在 ChronoTrace 平台层：
person_code 只是 fields_config 中的一个字段名，不是核心模型的一部分。
```

### 7.3 P3.2 应重命名和重写

原先的“人员主数据 / 人工匹配流程”方向应暂停并重审。

更合理的方向是：

```text
P3.2：无稳定实体标识导入治理
```

它应该回答的是通用问题：

```text
当任意 schema 的导入数据缺少稳定 identity 时，系统该如何处理？
```

而不是人员专属问题：

```text
没有员工号时，如何匹配同一个人？
```

### 7.4 通用原则

无稳定 identity 时，ChronoTrace 应遵循：

- 不自动模糊匹配直接入库。
- 不把同名、同简称、同地址、同部门等弱特征当成稳定身份。
- 不让行号型 `system_code` 用于周期性导入。
- 优先阻断并提示用户补充稳定编号。
- 如果未来做人工确认，也应设计成通用 `EntityResolution`，而不是 `PersonMaster`。
- 人工确认结果必须可审计、可撤销、可解释。

## 8. 导入身份治理边界

### 8.1 稳定 identity 是硬前提

ChronoTrace 的时态追踪依赖稳定 identity。

如果 identity 不稳定，会出现：

```text
第一次导入：张伟 -> Entity A
第二次导入：张伟 -> Entity B
第三次导入：张伟 -> Entity C
```

结果是历史生命周期断裂，系统无法回答：

```text
这个实体过去发生过什么？
本次导入是新增还是更新？
缺失记录应保留还是终止？
```

因此，identity 不是可选体验项，而是系统可信度门槛。

### 8.2 行号型 `system_code`

行号型 `system_code` 只适合一次性新表接入。

它不适合：

- 周期型表
- 每月导入
- 外部 Excel 行顺序可能变化的场景
- 需要长期追踪同一实体的场景

原因：

```text
第一月第 5 行不一定是第二月第 5 行。
```

如果用行号作为长期 identity，就会制造静默错配。

### 8.3 复合 identity

复合 identity 是当前系统正确方向之一。

例如：

```text
公司编码 + 员工号
仓库 + 货架 + 货位
供应商编码 + 合同编号
项目编号 + 阶段编号
```

系统内部可以继续生成一个稳定 `business_code`：

```text
C01|E001
```

用户界面可以展示更友好的 `display_code`：

```text
C01 / E001 / 张伟
```

这正是 `business_code` 与 `display_code` 拆分的价值。

## 9. 功能取舍原则

判断一个新功能是否属于 ChronoTrace 核心，可以问 8 个问题：

| 问题                              | 如果答案是“是”      |
| ------------------------------- | ------------- |
| 它是否围绕长期存在的业务实体？                 | 更可能属于核心       |
| 它是否增强 `Entity` 的时态追踪能力？         | 更可能属于核心       |
| 它是否让 ChangeSet 更可靠、更可审计？        | 更可能属于核心       |
| 它是否改善稳定 identity 的配置、诊断或展示？     | 更可能属于核心       |
| 它是否让导入、审批、回滚链路更可信？              | 更可能属于核心       |
| 它是否引入某个领域专属主数据模型？               | 需要警惕          |
| 它是否需要复杂交易、流程或报表引擎？              | 多半不属于当前核心     |
| 它是否能用普通 schema 字段表达，而不必写平台专属模型？ | 优先用 schema 表达 |

简化规则：

```text
增强时态实体链路的，优先考虑。
替代某个垂直业务系统的，默认拒绝或外置。
```

## 10. Bug 修复排序原则

接下来如果先修 bug，建议按“是否影响核心链路可信度”排序。

### P0：影响可信度和安全性的 bug

优先修：

- 导出绕过敏感字段脱敏。
- 导入 identity 诊断漏报、误报或允许重复通过。
- ChangeSet 应用后时态记录重叠、断裂或无法回滚。
- 审批状态错误导致未审批变更直接生效。
- 权限绕过导致 viewer 能修改或导出敏感数据。
- `display_code` 把敏感字段明文展示或导出。

这些问题会破坏系统可信度，应该优先于新功能。

### P1：影响主流程可用性的 bug

随后修：

- Excel 映射体验导致导入失败但错误不可理解。
- 缺失策略文案或行为不清，用户误选导致错误终止。
- 当前视图、生命周期、变更详情展示不一致。
- 审批人列表为空时缺少清晰处理路径。
- 复合 identity 在某些入口没有自动填充或展示。

这些问题影响用户能否正确使用系统。

### P2：体验和效率问题

再处理：

- 字段排序、搜索、分页体验不足。
- 展示模板配置体验优化。
- 统计卡片和趋势图增强。
- 大表性能优化。
- 常用视图保存。

这些可以提升产品质感，但不应压过可信度问题。

## 13. 最终结论

ChronoTrace 的核心不是“人员清单”，也不是“万能表格”。

它的核心是：

```text
现实实体
    -> 稳定业务身份
    -> 时态状态快照
    -> 变更批次
    -> 审批、回滚、审计
```

人员、资产、合同、供应商、账号、项目、风险事项都可以是现实实体。

但平台层必须保持通用：

```text
平台认识 Entity。
平台不内置 Person。
平台治理 identity。
平台不替业务系统猜测身份。
```

这条边界应作为后续需求判断和 bug 修复排序的基础。
